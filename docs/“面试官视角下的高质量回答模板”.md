# 1. ReAct 怎么实现？

## 面试回答（推荐）

我实现的是一个简化版 ReAct Agent。

核心思想是：



> LLM 不直接输出最终答案，而是先推理（Reason），再决定调用什么工具（Act），最后根据工具结果继续推理。

整体流程：



```
用户问题
   ↓z
LLM 生成 Thought
   ↓
判断是否需要 Tool
   ↓
调用 Tool
   ↓
Tool Result 回填上下文
   ↓
LLM 再次推理
   ↓
输出 Final Answer
```



在工程实现上：

-  Agent 维护 message history 
-  Tool 使用 `@tool` 装饰器注册 
-  自动生成 function schema 
-  Tool 执行结果重新注入 prompt 
-  通过 while loop 控制 ReAct 循环 

我还加了：

-  最大 step 限制 
-  Tool 超时保护 
-  Tool 异常吞噬 
-  SSE 流式输出 

避免 Agent 死循环。

------

## 面试加分句

> ReAct 本质上是“LLM 驱动的状态机”。

这句话很加分。

# 2. Tool Calling 怎么做？

## 推荐回答

我的 Tool Calling 参考 OpenAI function calling 设计。

核心是：

-  使用 `@tool` 
-  自动读取函数签名 
-  基于 Pydantic 生成 JSON Schema 
-  注入到 LLM tools 参数中 

例如：



```
@tool
def search_knowledge(query: str):
    ...
```



会自动生成：



```
{
  "name": "search_knowledge",
  "parameters": {
    "type": "object",
    "properties": {
      "query": {"type": "string"}
    }
  }
}
```



LLM 返回 tool_call 后：

-  Agent 解析 function name 
-  执行对应函数 
-  Tool Result 注入 message history 
-  再次调用 LLM 

这样实现：

> “LLM 决策 + Python 执行”

的闭环。

    ↓

# 5. 为什么用混合检索？

## 推荐回答

单纯向量检索在中文场景有两个问题：

-  专有名词召回不稳定 
-  精确关键词容易丢 

例如：



```
反洗钱法第23条
```



Embedding 不一定能稳定命中。

所以我采用：

# Hybrid Search



```
Dense Vector + BM25
```



------

## Dense

负责：

-  语义理解 

------

## BM25

负责：

-  关键词精确匹配 

------

## 最终：



```
score =
0.7 * dense
+
0.3 * sparse
```



召回效果明显更稳定。

# 6. Chunk 怎么切？

## 推荐回答

中文不能直接按：



```
split("\n\n")
```



因为：

-  中文段落不明显 
-  长句很多 

所以我做了：

# 递归切分

优先级：



```
段落
↓
换行
↓
句号
↓
问号
↓
逗号
↓
空格
↓
单字符
```



同时：

-  支持 overlap 
-  控制 chunk_size 
-  长 chunk 二次切分 

这样能：

-  保持语义完整 
-  提高 Recall
